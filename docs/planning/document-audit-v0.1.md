# 욕망백서 문서 충돌 / 중복 / 누락 점검 v0.1

## 1. 목적

- 현재까지 작성된 문서를 제품 기준값 관점에서 점검한다.
- 충돌, 중복, 누락 가능성을 정리하고 이후 정리 우선순위를 정한다.

## 2. 전체 판단

현재 문서 체계는 충분히 풍부하고, 제품 방향도 비교적 일관적이다.  
다만 문서 수가 많아지면서 아래 세 가지 관리 포인트가 생겼다.

- 같은 내용을 여러 문서에서 반복함
- 일부 문서는 버전명이 최신 상태와 맞지 않음
- 구현 직전 필요한 세부 기준 중 아직 빠진 항목이 있음

## 3. 현재 기준으로 충돌 여부 점검

### 3-1. 포인트 정책

현재 일관된 기준:

- 새 DM 요청 `90P`
- 거절 시 `45P 환불`
- 24시간 미응답 만료 시 `90P 전액 환불`
- 차단 시 환불 없음
- 여성 가입 시 `270P`

판단:

- 큰 충돌 없음
- 현재 문서 대부분이 이 숫자를 기준으로 맞춰져 있음

### 3-2. DM 상태값

현재 기준:

- `pending`
- `agreed`
- `declined`
- `expired`
- `blocked`
- `closed`

판단:

- 상태도, SQL, 정책 문서가 대체로 일치함
- `closed`의 실제 사용 시점은 아직 조금 추상적이라 구현 전 정의를 더 좁히는 것이 좋음

### 3-3. 플랫폼 전략

현재 기준:

- 초기 출시 `PWA / 모바일 웹`
- 앱은 후행 검토

판단:

- 포지셔닝, 정책, 시장조사 문서와 일관됨

### 3-4. 프로필 원칙

현재 기준:

- 프로필은 판단 도구
- 사진보다 취향, 자기소개, 최근 글 우선

판단:

- UX, 하이파이, 정보구조 문서 전반에서 일관됨

## 4. 중복이 많은 영역

### 4-1. 포인트 / 환불 설명

중복 문서:

- PRD
- 정책 문서
- 포인트/환불 정책
- 포인트 UX 문서
- 와이어카피
- FAQ
- 정책 UI 매핑

판단:

- 중복 자체는 필요하지만, 숫자가 바뀌면 같이 수정할 문서가 많다

권장:

- 기준 원문은 `legal/point-refund-policy-v0.1.md`로 두고
- 나머지는 요약/노출 문구로 관리

### 4-2. 브랜드 설명

중복 문서:

- 브랜드 포지셔닝
- 디자인 시스템
- 카피 라이브러리
- 네이밍/카피 확장안

판단:

- 역할은 다르지만 메시지 중복이 있다

권장:

- 브랜드 정의 원본은 `brand-positioning-v0.2.md`
- 디자인 문구 기준은 `design-system-v1.md`
- 실제 서비스 문구는 `copy-library-v0.1.md`

### 4-3. 화면 설명

중복 문서:

- screens
- wireframes
- hifi ui spec
- wire copy

판단:

- 자연스러운 중복
- 다만 각 문서 역할이 분명해야 함

권장 역할:

- `screens`: 화면 목록과 기능 개요
- `wireframes`: 구조
- `hifi ui spec`: 시각/배치 디테일
- `wire copy`: 실제 텍스트

## 5. 버전/명칭 정리 필요 항목

### 5-1. 브랜드 포지셔닝

- `brand-positioning-v0.2.md`가 현재 최신
- README와 참조 문서에서 더 이상 `v0.1`로 부르지 않도록 주의 필요

### 5-2. README 인덱스

- 현재는 최신 상태로 보이지만 이후 문서 추가가 계속 되므로 가장 먼저 누락되기 쉬운 파일임

### 5-3. 초기 문서 버전

- `prd-v0.1.md`, `policy-v0.1.md` 등은 내용이 여러 번 반영되었음
- 시점이 더 지나면 `v0.2`로 올려서 정리하는 것이 좋음

## 6. 구현 전 아직 부족한 영역

### 6-1. RLS 정책 상세

현재 상태:

- 방향은 있음
- 실제 정책 SQL 문서는 없음

필요 이유:

- Supabase 구현 시 가장 민감한 부분 중 하나

### 6-2. 포인트 충전 운영 상세

현재 상태:

- 무통장/가상계좌 중심 흐름은 있으나 실제 운영 예외 처리 기준은 얕음

필요 이유:

- 결제 오류, 입금자명 불일치, 중복 입금 대응이 필요함

### 6-3. 탈퇴 / 데이터 삭제 상세

현재 상태:

- 개인정보처리방침 초안은 있으나 탈퇴 처리 상세는 부족

필요 이유:

- 운영 문의와 법률 문서 보강 필요

### 6-4. 관리자 권한 체계

현재 상태:

- 1인 운영 기준 문서는 충분
- 다중 운영자 또는 권한 분리 기준은 없음

판단:

- 지금은 없어도 되지만 추후 확장 시 필요

## 7. 구현 전 우선 정리 권장 순서

1. `RLS 정책 설계 문서`
2. `포인트 충전 운영 상세 문서`
3. `탈퇴 / 데이터 삭제 정책 문서`
4. 주요 기획 문서 `v0.2` 정리

## 8. 지금 문서 체계의 강점

- 브랜드와 UX 방향이 선명함
- 포인트/환불 정책이 비교적 명확함
- 운영 시나리오가 생각보다 잘 정리됨
- PWA 전략이 흔들리지 않음

## 9. 지금 문서 체계의 리스크

- 양이 많아지면서 신규 합류자가 길을 잃을 수 있음
- 같은 숫자를 여러 문서에서 반복해 수정 누락 위험이 있음
- 일부 문서는 실제 구현 직전 세부성이 아직 부족함

## 10. 최종 판단

지금 문서 상태는 `기획 부족` 단계가 아니라 `정리와 연결` 단계다.

즉, 새 문서를 무한히 늘리는 것보다  
핵심 기준 문서를 중심으로 업데이트 체계를 유지하는 것이 더 중요하다.
