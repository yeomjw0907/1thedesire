# 성향·태그 배치 UX 비교 (프로필 vs 글 태그)

다른 데이팅 서비스와 비교해, **BDSM 역할 = 프로필**, **FWB/데이팅 등 = 글 쓸 때 태그** 방안이 UX적으로 괜찮은지 정리한 문서.

---

## 1. 시장에서의 일반 패턴

| 서비스 | 관계 유형 / "찾는 것" (FWB, 캐주얼 등) | BDSM·역할 | 비고 |
|--------|----------------------------------------|------------|------|
| **Tinder** | 프로필 관계 태그 (6종: 장기/단기 등) 1개 | - | 프로필 단일 선택 |
| **Bumble** | 프로필 "Looking for" 등 | - | 프로필 |
| **Feeld** | 프로필 "Relationship types" + Desires 30개+ | 프로필 Desires에 Dom/Sub 등 포함 | 모두 **프로필**에 한 번 설정, 검색·필터에 사용 |
| **Grindr** | 프로필 "My Tags" (최대 8개) | 태그로 표현 | **프로필 태그**만, 클릭 시 같은 태그 유저 검색 |
| **Hinge** | 프로필 프롬프트 (3개) | - | 프로필 |
| **일반 트렌드** | "Looking for" / 관계 목표 = **프로필 수준**에서 1회 설정 → 매칭 품질용 | 역할·취향도 **프로필**에 두고 검색·필터에 활용 | **per-conversation / per-post 선택지는 거의 없음** |

**정리**: 대부분 데이팅 앱은 **FWB, 데이팅, 캐주얼, LTR** 등을 **프로필**에 두고, "이 사람이 뭘 원하는지" 한 번에 보이게 하고 검색·매칭에 씁니다.

---

## 2. 우리 서비스 구조와의 차이

- **일반 데이팅 앱**: 발견(discovery) 단위 = **프로필 카드** (스와이프/좋아요)
- **욕망백서**: 발견 단위 = **피드의 글(분위기)** → "분위기 먼저 보고, 대화는 그 다음"

그래서:

- 다른 앱: "이 **사람**이 FWB 원해" → 프로필 1곳에 두는 게 자연스러움
- 우리: "이 **글(분위기)**이 FWB 느낌이야" → **글 단위**에 태그를 두는 것도 자연스러움

즉, **발견 단위가 글(포스트)**이기 때문에 "FWB/데이팅을 글 태그로" 두는 건 **서비스 구조와 맞는 선택**입니다.

---

## 3. UX 관점에서의 장단

### 3.1 FWB·데이팅을 **글(포스트) 태그**로 둘 때

**장점**

- **글마다 맥락 구분 가능**: 같은 사람이 "이번엔 대화 위주", "이번엔 만남 위주" 등 **글마다 다른 분위기**를 낼 수 있음
- **가입 부담 감소**: 가입 시에는 BDSM 역할(Dom/Sub/Switch) 1개만 → 단순
- **분위기 중심 컨셉과 일치**: "이 글의 분위기"를 태그로 표현하는 구조와 잘 맞음
- **검색 가능**: 나중에 "FWB 글만 보기", "감성 연애 글만 보기" 같은 **글 단위 필터** 구현 가능

**단점 / 리스크**

- **시장 관행과 다름**: 다른 앱은 대부분 "찾는 것"을 프로필에 둠 → 처음 쓰는 유저가 "어디서 FWB 고르지?"라고 헷갈릴 수 있음
- **검색 시**: "FWB 원하는 **사람** 찾기"가 목적이면, 글 태그 기반 검색만으로는 "이 사람이 전반적으로 FWB인지" 한눈에 안 보일 수 있음 (여러 글을 봐야 할 수 있음)

### 3.2 FWB·데이팅을 **프로필**에 둘 때 (시장 대부분)

**장점**

- **익숙한 UX**: "프로필에 관계 목표 한 번 쓰기"에 사용자가 익숙함
- **검색·매칭 단순**: "FWB 원하는 사람" 필터 한 번에 가능
- **한 화면에 정리**: 프로필 한 번 보면 "역할 + 찾는 관계" 다 보임

**단점**

- 우리는 **글 피드가 메인**이라, "이 사람 프로필에만 FWB 있고, 이번 글은 다른 맥락"인 경우를 구분하기 어려움
- 가입 시 선택지가 다시 늘어남 (역할 + 관계 유형)

---

## 4. 결론: 우리 방안이 UX적으로 괜찮은가?

**괜찮다**고 보는 근거:

1. **발견 단위가 글**이므로, "이 글의 맥락(FWB/데이팅 등)"을 **글에 붙이는 건 우리 구조에 맞음**.
2. **BDSM 역할만 프로필**로 두는 건 시장과 동일한 방향(역할 = 상대적으로 고정된 정체성).
3. **FWB/데이팅을 글 태그**로 두면, 다른 앱과 다르지만 **우리 컨셉(분위기·글 중심)**과는 잘 맞고, 정당한 설계 선택으로 설명 가능.

**보완하면 좋은 점** (나중 단계):

- **온보딩/도움말**: "원하는 관계(FWB, 데이팅 등)는 **글 쓸 때** 태그로 달 수 있어요" 한 줄 안내 → "프로필에 없네?" 하는 혼란 감소.
- **검색**: "FWB 포함 글 보기" 등 **글 태그 기반 필터**를 넣으면, "FWB 원하는 사람 찾기" 니즈도 글 단위로 충족 가능.
- (선택) **프로필 요약**: "최근 글에 자주 단 태그"를 프로필에 요약 노출하면, "이 사람이 대체로 뭘 원하는지" 한눈에 보이게 할 수 있음. (완전 필수는 아님)

---

## 5. 요약 표

| 항목 | 시장 대부분 | 우리 제안 | UX적 판단 |
|------|-------------|-----------|-----------|
| BDSM 역할 (Dom/Sub/Switch) | 프로필 | **프로필 (가입 시 1개)** | 시장과 동일, 적절 |
| FWB, 데이팅, 대화 위주 등 | 프로필 | **글(포스트) 태그** | 우리는 **글 중심**이라 글 태그가 구조에 맞음. 다소 비주류이지만 일관된 선택 |
| 사용자 학습 | "프로필에 다 쓴다"에 익숙 | "역할은 프로필, 관계/분위기는 글에 태그" | 도움말/온보딩 한 줄이면 보완 가능 |

**최종**: 다른 데이팅 서비스와 비교했을 때, **BDSM은 프로필, FWB·데이팅은 글 쓸 때 태그**로 두는 방안은 **UX적으로 타당한 설계**이며, 우리 서비스(분위기·글 중심)와도 맞습니다.
