# 욕망백서 추후 앱 출시 제약 조건 v0.1

## 1. 목적

- 향후 욕망백서를 앱으로 출시할 가능성에 대비해 제약 조건을 정리한다.
- 지금부터 어떤 기능과 표현이 앱 전환 시 문제가 될 수 있는지 기록한다.

## 2. 기본 원칙

- 현재 MVP는 앱스토어 제출을 기준으로 설계하지 않는다.
- 다만 추후 앱 전환 가능성을 완전히 닫지는 않는다.
- 따라서 지금부터 `앱 전환 시 수정이 필요한 항목`을 분리해 관리한다.

## 3. 앱 전환 시 위험한 요소

### 3-1. 메타데이터

- BDSM, FWB, 성적 욕망, 성적 대화 같은 직접적 표현
- 노골적인 스크린샷 문구
- 성인 목적의 만남을 직접 강조하는 카피

### 3-2. 기능 구조

- 성적 대화 자체를 핵심 가치로 드러내는 구조
- 앱 내에서 노골적인 성적 태그나 소개글 노출
- 익명 랜덤 채팅처럼 보이는 UX
- 과도한 성적 만족 중심 탐색 기능

### 3-3. 시각 요소

- 노출이 강한 이미지
- 자극적인 썸네일
- 과하게 유혹적인 광고성 표현

## 4. 앱으로 갈 때 조정해야 할 항목

- 메인 카피를 `취향 커뮤니티`, `관계 탐색`, `성인 간 소통` 수준으로 낮추기
- 스토어 스크린샷은 안전, 커뮤니티, 프로필, 필터 중심으로 구성하기
- 민감한 태그와 표현은 기본 비노출 또는 웹 전용으로 분리하기
- 신고/차단/가이드라인을 앱 내에서 더 강하게 전면 노출하기

## 5. 앱에서도 살릴 수 있는 기능

- 피드 기반 탐색
- 프로필 중심 구조
- 취향 태그의 순화 버전
- 대화 요청과 수락 구조
- 신고, 차단, 운영 가이드

## 6. 앱에서 웹으로 넘기는 것이 유리한 기능

- 민감한 콘텐츠 열람
- 직접적인 성적 표현이 담긴 피드
- 고위험 프로필 필드
- 정책상 민감한 커뮤니티 영역

## 7. 지금부터 관리해야 할 태깅 기준

향후 앱 전환을 고려하면 기능은 같아도 콘텐츠 레벨 분류가 필요하다.

### 제안 분류

- `safe`: 앱/웹 공통 노출 가능
- `sensitive`: 웹 우선 노출
- `restricted`: 웹 한정 또는 별도 검토

## 8. 출시 판단 게이트

추후 앱 출시를 검토하려면 아래 조건이 필요하다.

1. 핵심 기능 중 앱에서 빼도 되는 기능이 구분되어 있어야 한다
2. 메타데이터와 카피를 순화한 별도 브랜딩 버전이 있어야 한다
3. 신고, 차단, 운영정책이 충분히 정리되어 있어야 한다
4. 유저 피드백상 앱 수요가 분명해야 한다

## 9. 최종 판단

앱 출시는 `지금 당장 할 일`이 아니라 `조건이 맞을 때 선택할 옵션`이다.

지금은 웹/PWA에서 제품을 제대로 만들고, 앱은 그 다음에 최소 리스크 구조로 분기하는 것이 맞다.
