## 1-review.md 기준 개선 가능 여부

### ✅ 코드/설정만으로 가능 (우선 적용 권장)

| 항목 | 내용 | 가능 여부 |
|------|------|-----------|
| **OAuth `next` 오픈 리다이렉트** | `auth/callback`의 `next` 파라미터를 내부 경로 허용 목록으로만 제한 | ✅ 가능. 허용 목록 검증 추가하면 됨. |
| **한글/메타데이터 인코딩** | UTF-8 통일, 깨진 문자열 복구, layout/login 등 메타데이터 점검 | ✅ 가능. 파일 인코딩·문자열·title/description 점검 후 수정. |
| **접근성: 확대 제한 제거** | `userScalable: false` 제거 | ✅ 가능. layout viewport 메타 수정. |
| **DM 과금 정책 보강** | 환불 규칙 외 신고/차단·응답 기한·안심 문구 등 구조적 노출 | ✅ 부분 가능. 문구·위치 보강은 가능. |

### ⚠️ DB/인프라·제품 결정 필요

| 항목 | 내용 | 가능 여부 |
|------|------|-----------|
| **포인트 차감/환불 트랜잭션화** | RPC 또는 SQL 함수로 차감·방 생성·로그를 한 트랜잭션으로 | ⚠️ 가능하나 DB 함수 설계·마이그레이션 필요. |
| **가입 완료 원자 처리** | 프로필·포인트·자동 게시글을 DB 함수 등으로 원자 처리 | ⚠️ 위와 동일. |
| **결제(충전) 플로우** | 입금 접수·입금자명·승인 큐·PG 또는 입금 신청 테이블 | ⚠️ 제품/운영 정책 결정 후 설계·구현 필요. |
| **관리자 RBAC** | 이메일 비교 대신 profiles 또는 admin role 테이블 기반 | ⚠️ 스키마·권한 모델 설계 후 구현. |

### 📋 제품/기획 결정 후 진행

| 항목 | 내용 |
|------|------|
| 여성 보너스·무료 수락 | 품질 기반 인센티브로 전환 등 정책 변경 필요. |
| 신뢰 지표 (본인확인·응답률 등) | 수집 항목·노출 범위 기획 필요. |
| 2차 온보딩 (사진·키워드 등) | 기획·디자인 후 구현. |
| ESLint flat config + CI | 설정 추가 및 CI 파이프라인 구성 필요. |

## 2~7 (STI 관련) 문서

- **2** 법적 리스크·허용 범위  
- **3** 정책/약관 반영안  
- **4** DB 스키마·접근권한  
- **5** 운영 검수 프로세스  
- **6** UX 문구·노출 리스크  
- **7** 수익모델 연결  

→ 모두 **「최근 STI 검사 확인」이라는 신규 기능**을 넣기 위한 설계 문서입니다.  
**현재 앱에 없는 기능**을 추가하는 것이므로, “기존 서비스 개선”이 아니라 **신규 기능 구현** 단계에서 참고하는 용도입니다.  
구현 가능 여부는 “STI 기능을 도입할지” 제품 결정이 선행된 뒤, 2~7 문서대로 정책·DB·운영·UX를 맞추면 됩니다.

## 정리

- **지금 당장 진행하기 좋은 것**: 1-review.md 중  
  - OAuth `next` 검증  
  - 한글/메타데이터 인코딩·노출 점검  
  - viewport `userScalable` 제거  
  - DM 정책 문구·노출 보강  
- **DB/스키마 작업 후 진행**: 포인트·가입 원자 처리, 관리자 RBAC.  
- **제품/기획 결정 후**: 결제 플로우, 여성 보너스 정책, 신뢰 지표, 2차 온보딩, CI/ESLint.  
- **STI(2~7)**: 신규 기능 설계 문서이므로, 기능 도입 결정 후 2~7을 참고해 구현 가능 여부·범위를 정하면 됩니다.
