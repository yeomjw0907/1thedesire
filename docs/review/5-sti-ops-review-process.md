# STI 기능 운영 검수 프로세스 검토안

## 목적
- `최근 STI 검사 확인` 기능을 수동 또는 반수동 검수 방식으로 운영할 때 필요한 프로세스를 정리한다.
- 핵심은 `오탐`, `위조`, `과도한 인력 투입`, `민감정보 과열람`을 줄이는 것이다.

## 결론
- 이 기능은 자동 승인보다 `제한된 운영자 수동 검수`가 맞다.
- 다만 운영자가 원본 파일을 오래 보거나 많이 보는 구조는 위험하다.
- 따라서 검수는 짧고 표준화된 체크리스트 기반으로 끝내야 한다.

## 1. 권장 운영 플로우

### 사용자 플로우
1. 사용자가 기능 소개를 읽음
2. 민감정보 수집·이용 동의
3. 공개 동의 선택
4. 제출
5. 검수 대기
6. 승인 또는 반려
7. 승인 시 배지 노출
8. 만료 전 재검증 안내

### 운영자 플로우
1. 대기열 확인
2. 제출 건 열람
3. 체크리스트 검증
4. 승인/반려/추가 제출 요청
5. 감사 로그 기록
6. 원본 즉시 삭제 확인

## 2. 검수 체크리스트

### 최소 체크 항목
- 제출자가 본인인지 식별 가능한 최소 단서가 있는가
- 최근 검사일이 확인 가능한가
- 회사가 요구한 검증 형식에 부합하는가
- 위조 정황이 없는가
- 유효기간 산정이 가능한가

### 검수자가 판단하면 안 되는 항목
- 이용자의 성생활 패턴 추정
- 이용자 위험도 등급화
- 세부 의학적 판독
- 의료적 조언

## 3. 승인 기준

### 승인 가능 조건
- 최근 검사 사실을 확인할 수 있음
- 제출 형식이 내부 기준에 맞음
- 검수 항목 누락이 없음

### 반려 조건
- 자료 훼손
- 식별 불가
- 날짜 확인 불가
- 타인 자료 의심
- 허용 형식 아님

### 제재 검토 조건
- 명백한 위조
- 타인 자료 반복 제출
- 반려 후 반복 허위 시도

## 4. 운영 정책 세부안

### SLA 권장안
- 1차 응답 목표: 24시간 이내
- 승인/반려 목표: 48시간 이내

### 재신청 정책
- 단순 반려: 즉시 재신청 가능
- 위조 의심 반려: 추가 확인 후 재신청 허용
- 명백한 허위: 일정 기간 제한 또는 계정 제재

### 철회 정책
- 이용자는 언제든 공개 철회 가능
- 철회 시 즉시 배지 비노출
- 철회 후에도 감사 로그는 보존 가능

## 5. 운영 조직 설계

### 권장 역할 분리
- 일반 CS 운영자
- STI 검수 운영자
- 최종 승인 권한자

### 이유
- 민감정보를 모든 운영자가 볼 필요가 없다.
- 조회 인원을 줄여야 침해 가능성이 줄어든다.

## 6. 관리자 화면에 필요한 요소
- 제출 대기 수
- 제출 시각
- 처리 SLA 남은 시간
- 승인/반려 버튼
- 반려 사유 템플릿
- 만료 예정 목록
- 박탈 이력
- 감사 로그 보기

## 7. 감사 로그 기준

### 반드시 남길 로그
- 누가 열람했는지
- 누가 승인/반려했는지
- 언제 처리했는지
- 공개 설정이 언제 바뀌었는지
- 원본 삭제가 완료됐는지

### 이유
- 민감정보 기능에서 로그는 사실상 통제 장치다.

## 8. 운영 리스크

### 가장 큰 리스크
- 운영자 과열람
- 기준 없는 승인
- 반려 사유 불명확
- 처리 지연
- 원본 삭제 누락

### 대응
- 체크리스트 표준화
- 반려 사유 템플릿화
- 처리 시간 SLA 관리
- 정기 감사

## 9. MVP 운영 권장안
- 일단 수동 검수
- 일단 하루 처리량 상한 설정
- 일단 승인 기준 보수적으로 운용
- 원본 장기 보관 금지

## 10. 권장 결론
- 운영 프로세스가 준비되지 않으면 기능을 열지 않는 게 맞다.
- 특히 `배지는 달아주고 운영은 나중에` 방식은 금지해야 한다.
