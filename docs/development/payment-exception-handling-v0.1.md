# 욕망백서 포인트 충전 / 입금 예외 처리 문서 v0.1

## 1. 목적

- 포인트 충전과 입금 과정에서 발생할 수 있는 예외 상황을 정리한다.
- 운영자가 어떤 기준으로 대응해야 하는지 기본 정책을 만든다.

## 2. 기본 원칙

- 결제/입금 이슈는 빠르게 확인한다
- 실제 입금 여부와 시스템 반영 여부를 분리해서 본다
- 포인트 반영과 환불은 반드시 로그로 남긴다
- 운영 예외는 수동 조정하더라도 기록이 있어야 한다

## 3. 정상 흐름

1. 사용자가 충전 안내 확인
2. 입금 또는 결제 진행
3. 시스템에서 결제 이벤트 생성
4. 포인트 지급
5. 거래 내역 기록

## 4. 예외 상황 목록

### 4-1. 입금했는데 포인트가 안 들어옴

확인할 것:

- 실제 입금 여부
- 입금 시간
- 입금 금액
- 입금자명
- 시스템 이벤트 생성 여부

조치:

- 입금 확인 시 수동 반영 가능
- `payment_events`와 `point_transactions` 로그 남김

### 4-2. 입금자명이 다름

확인할 것:

- 회원 정보와 입금자명 차이
- 문의 접수 내용

조치:

- 확인 가능한 경우 수동 반영
- 반복 발생 시 안내 문구 강화 필요

### 4-3. 중복 입금

확인할 것:

- 동일 유저 동일 금액 중복 여부
- 실제 중복 입금인지

조치:

- 추가 포인트 지급 또는 환불 여부를 운영 기준으로 결정
- 반드시 수동 조정 로그 남김

### 4-4. 시스템은 성공인데 포인트가 중복 반영됨

확인할 것:

- payment event 중복
- point transaction 중복

조치:

- 중복 지급분 회수 또는 수동 조정
- 재발 방지를 위해 idempotency 점검

### 4-5. 결제 실패인데 포인트가 들어감

확인할 것:

- 결제 이벤트 상태
- 실제 입금 여부

조치:

- 사실상 미입금이면 포인트 회수 검토
- 이미 사용된 경우 운영자 판단 필요

### 4-6. 환불이 중복 반영됨

확인할 것:

- `declined`, `expired` 이벤트 중복 발생 여부
- 동일 room 기준 환불 로그 중복 여부

조치:

- 수동 조정
- 서버 액션 중복 실행 방지 검토

## 5. 운영 응답 기준

### 사용자 안내 톤

- 짧고 명확하게
- 먼저 사실 확인
- 감정적 표현 금지

예:

- 입금 내역을 확인 중입니다
- 확인 후 순차적으로 반영해드리겠습니다
- 현재 거래 상태를 기준으로 안내드리고 있습니다

## 6. 운영자 확인 체크리스트

- 실제 입금이 있었는가
- 시스템 이벤트가 생성됐는가
- 포인트 거래 로그가 남았는가
- 이미 수동 조정된 적이 있는가
- 같은 문제가 반복되는가

## 7. 수동 조정 기준

### 허용되는 수동 조정

- 미반영 입금 보정
- 중복 지급 회수
- 운영 오류 수정

### 수동 조정 시 필수 기록

- 대상 유저
- 조정 금액
- 이유
- 관련 payment event 또는 reference
- 처리 시각

## 8. UI / 문구 보완 포인트

- 충전 화면에 입금자명 안내를 명확히 둘 것
- 중복 입금/오입금 문의 경로를 제공할 것
- 충전 내역과 환불 내역을 분리해서 보여줄 것

## 9. 추후 고도화 포인트

- idempotency key 도입
- 결제 이벤트 자동 검증
- 운영자 수동 조정 UI 고도화

## 10. 최종 판단

포인트 충전 예외 처리는 결제 기능 자체보다  
`운영자가 예외를 얼마나 빨리 정리할 수 있는가`가 더 중요하다.

기준이 없으면 신뢰가 빠르게 무너진다.
