# 욕망백서 RLS 정책 설계 v0.1

## 1. 목적

- Supabase Row Level Security 기준을 문서화한다.
- 사용자, 운영자, 시스템 액션이 어떤 데이터에 접근할 수 있는지 정리한다.

## 2. 기본 원칙

- 기본값은 `거부`다
- 사용자는 본인 데이터와 본인이 참여한 데이터만 접근 가능해야 한다
- 운영자 권한은 일반 사용자와 분리한다
- 상태값과 차단 관계를 고려한 접근 제어가 필요하다

## 3. 역할 정의

### 일반 사용자

- 인증된 사용자
- 본인 프로필, 게시글, 대화, 포인트 이력 일부 조회 가능

### 운영자

- 신고 검토
- 게시글 숨김
- 제재 이력 확인
- 포인트 및 환불 예외 처리

### 시스템 액션

- 가입 보너스 지급
- 자동 소개글 생성
- DM 만료 처리
- 환불 반영

## 4. 테이블별 정책 방향

## 5. profiles

### 조회

- 일반 사용자는 공개 가능한 프로필을 조회 가능
- 본인은 본인 프로필 전체 조회 가능
- 운영자는 전체 조회 가능

### 수정

- 본인은 본인 프로필만 수정 가능
- 운영자는 상태값 조정 가능

### 삭제

- 일반 사용자는 직접 삭제보다 탈퇴 프로세스를 통해 처리

### 추가 메모

- 향후 프로필 공개 범위 제어가 생기면 조회 정책이 세분화되어야 함

## 6. posts

### 조회

- 일반 사용자는 `published` 상태만 조회 가능
- 작성자는 본인 글 전체 조회 가능
- 운영자는 `hidden`, `deleted` 포함 조회 가능

### 생성

- 인증된 사용자만 생성 가능
- 시스템은 자동 소개글 생성 가능

### 수정 / 삭제

- 작성자 본인만 가능
- 운영자는 숨김/삭제 상태 변경 가능

## 7. blocks

### 조회

- 본인이 건 차단 / 당한 차단 여부는 시스템 판단에 필요
- 일반 사용자에게는 본인 관련 차단 관계만 허용
- 운영자는 전체 조회 가능

### 생성

- 인증된 사용자는 본인 기준 차단만 생성 가능

### 수정 / 삭제

- 일반 사용자는 본인 차단 해제만 가능하도록 추후 설계 가능

## 8. chat_rooms

### 조회

- 발신자와 수신자만 해당 방 조회 가능
- 운영자는 전체 조회 가능

### 생성

- 인증된 사용자가 DM 요청 시 생성 가능
- 차단 관계가 있으면 생성 불가

### 상태 변경

- 수신자는 `pending -> agreed / declined` 가능
- 시스템은 `pending -> expired` 가능
- 차단 행위는 `pending/agreed -> blocked` 가능
- 운영자는 예외 처리 가능

## 9. consent_events

### 조회

- 해당 room 참여자만 조회 가능
- 운영자는 전체 조회 가능

### 생성

- 시스템 또는 서버 액션에서만 생성

## 10. messages

### 조회

- 참여자만 조회 가능
- 운영자는 신고 또는 검토 목적일 때 조회 가능

### 생성

- `chat_rooms.status = 'agreed'`일 때만 가능
- 차단 상태면 생성 불가

### 수정 / 삭제

- MVP에서는 편집 기능 없음
- 상태 변경은 운영 또는 시스템 액션으로만 제한

## 11. point_transactions

### 조회

- 본인 거래 내역만 조회 가능
- 운영자는 전체 조회 가능

### 생성

- 일반 사용자가 직접 생성 불가
- 시스템 또는 운영 액션만 가능

### 수정

- 일반 사용자 수정 불가
- 운영자는 예외 처리용 수동 조정 가능

## 12. payment_events

### 조회

- 일반 사용자는 본인 결제 결과 일부만 확인 가능
- 운영자는 전체 조회 가능

### 생성

- 시스템 액션만 생성

## 13. reports

### 조회

- 신고자는 본인 신고 내역 일부 확인 가능
- 운영자는 전체 조회 가능

### 생성

- 인증된 사용자는 신고 생성 가능

### 수정

- 일반 사용자는 수정 불가
- 운영자는 상태 변경 가능

## 14. moderation_actions

### 조회

- 일반 사용자 접근 불가
- 운영자만 조회 가능

### 생성

- 운영자만 생성 가능

## 15. 상태 / 관계 기반 제어 포인트

### 차단 관계

- DM 요청 생성 불가
- 메시지 작성 불가
- 일부 조회 결과에서 상호 비노출 처리 필요

### 계정 상태

- `restricted`: 일부 기능 제한
- `suspended`: 대부분 기능 제한
- `banned`: 접근 차단

### 채팅 상태

- `pending`: 조회만 가능, 메시지 작성 불가
- `agreed`: 메시지 작성 가능
- `declined`, `expired`, `blocked`, `closed`: 메시지 작성 불가

## 16. 구현 권장 방식

- RLS는 접근 여부만 담당
- 복잡한 상태 전이와 포인트 증감은 서버 액션에서 처리
- 운영자 권한은 별도 role claim 또는 secure function 기반으로 분리

## 17. 주의할 점

- RLS만으로 포인트 비즈니스 로직을 처리하려 하지 말 것
- 운영자 우회 권한은 최소화할 것
- 차단 관계는 조회와 생성 둘 다 막아야 함

## 18. 최종 판단

욕망백서의 RLS 핵심은 `누가 무엇을 볼 수 있는가`보다  
`누가 언제 어떤 상태의 데이터를 건드릴 수 있는가`를 명확히 하는 것이다.

특히 `chat_rooms`, `messages`, `point_transactions`, `reports`가 핵심이다.
