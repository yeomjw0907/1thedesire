# STI Verification Wireflow v0.1

> **카피 기준**: 사용자 노출 문구는 [recent-check-copy-system-v0.1.md](recent-check-copy-system-v0.1.md)를 따른다.
> 사용자 노출 메인명은 `최근검사 확인`이다.

## 목적
- `최근검사 확인` 기능을 화면 단위로 잘게 나누어 정리한다.
- 대상
- 사용자 플로우
- 관리자 플로우
- 상태별 예외 플로우

## 공통 원칙
- 검증 상태와 공개 여부는 분리
- 원본 제출물은 검수 완료 후 삭제
- 프로필 노출은 `verified + is_public + not expired` 조건일 때만 허용
- 사용자 노출 메인명은 `최근검사 확인`만 사용

## 상태 정의
- `none`
- `pending`
- `under_review`
- `verified`
- `rejected`
- `expired`
- `revoked`

## 사용자 플로우

### U1. 진입 화면
### 목적
- 기능의 의미와 주의사항을 먼저 이해시키는 화면

### 진입 경로
- 마이페이지
- 프로필 편집
- 신뢰 정보 섹션

### 표시 요소
- 제목: `최근검사 확인`
- 설명:
  - `최근 STI 검사 확인 정보를 프로필에 표시할 수 있습니다.`
  - `이 정보는 검사 시점 기준이며 현재 상태를 보증하지 않습니다.`
- 안내 포인트:
  - 어떤 정보가 공개되는지
  - 원본은 검수 후 삭제된다는 점
  - 언제든 공개 철회 가능하다는 점

### 버튼
- `인증 시작`
- `나중에`

### 다음 화면
- U2 동의 화면

## U2. 동의 화면
### 목적
- 민감정보 처리와 공개 여부에 대해 명확히 동의받는 화면

### 체크 항목
- `민감정보 수집·이용에 동의합니다`
- `검수 목적 처리에 동의합니다`
- `검증 완료 시 프로필에 공개하는 것에 동의합니다`

### 보조 문구
- `공개 동의는 이후에도 철회할 수 있습니다.`

### 버튼
- `다음`
- `취소`

### 검증
- 필수 동의 미체크 시 다음 단계 진입 불가

### 다음 화면
- U3 제출 화면

## U3. 제출 화면
### 목적
- 실제 검수 요청 데이터를 받는 화면

### 입력 항목
- `검사일`
- `검수용 이미지/파일 업로드`

### 표시 요소
- 허용 파일 형식
- 최대 파일 크기
- 검수 예상 시간
- 검수 완료 후 원본 삭제 안내

### 버튼
- `제출`
- `뒤로`

### 유효성 검사
- 검사일 필수
- 파일 필수
- 형식/크기 검증

### 서버 처리
- submission 생성
- badge 상태 `pending`
- audit log `submission_created`

### 다음 화면
- U4 제출 완료 화면

## U4. 제출 완료 화면
### 목적
- 사용자에게 검수 대기 상태를 명확히 알려주는 화면

### 표시 요소
- 상태 배지: `검수 대기`
- 안내 문구:
  - `제출이 완료되었습니다.`
  - `보통 24~48시간 내 검수가 완료됩니다.`
- 공개 여부 요약
- 검사일 요약

### 버튼
- `내 상태 보기`
- `프로필로 돌아가기`

### 다음 화면
- U5 상태 조회 화면

## U5. 상태 조회 화면
### 목적
- 사용자가 현재 검수 상태를 확인하는 화면

### 상태별 분기

#### `pending`
- 텍스트: `검수 대기 중`
- 설명: `아직 검수가 시작되지 않았습니다.`

#### `under_review`
- 텍스트: `검토 중`
- 설명: `운영자가 제출 내용을 확인 중입니다.`

#### `verified`
- 텍스트: `검증 완료`
- 검사일 표시
- 유효기간 표시
- 공개 여부 표시

#### `rejected`
- 텍스트: `반려됨`
- 반려 사유 표시
- 재제출 CTA 노출

#### `expired`
- 텍스트: `만료됨`
- 재검증 CTA 노출

#### `revoked`
- 텍스트: `철회됨`
- 재신청 CTA 노출

### 버튼
- 상태에 따라 달라짐
- `재제출`
- `공개 설정`
- `철회`

## U6. 공개 설정 화면
### 목적
- 검증 완료 후 프로필 배지 공개 여부를 설정하는 화면

### 노출 조건
- `verified` 상태일 때만 진입 가능

### 표시 요소
- 현재 공개 상태
- 프로필 미리보기
- 배지 문구 예시

### 토글
- `프로필에 공개`

### 버튼
- `저장`
- `취소`

### 저장 규칙
- `is_public = true/false`
- audit log `badge_visibility_updated`

## U7. 공개 중 프로필 화면
### 목적
- 실제 유저 프로필에 배지가 어떻게 보이는지 정의

### 표시 위치
- 닉네임/기본 정보 하단
- 신뢰 정보 섹션 내부

### 표시 요소
- 칩: `최근검사 확인`
- 서브텍스트:
  - `검사일 YYYY-MM-DD`
  - `YYYY-MM-DD까지 표시`

### 상호작용
- 클릭 시 U8 상세 모달

## U8. 배지 상세 모달
### 목적
- 과도한 신뢰 해석을 막는 설명 모달

### 표시 요소
- 제목: `최근검사 확인`
- 본문:
  - `이 프로필은 최근 STI 검사 확인 정보를 공개하고 있습니다.`
  - `검사 시점 기준의 정보이며 현재 상태를 보증하지 않습니다.`
  - `만남 전에는 대화와 보호수단 여부도 함께 확인해 주세요.`

### 버튼
- `닫기`

## U9. 반려 화면
### 목적
- 반려 시 재제출 동선을 분명히 하는 화면

### 표시 요소
- 상태: `반려됨`
- 반려 사유
- 재제출 가이드

### 버튼
- `다시 제출`
- `닫기`

### 다음 화면
- U3 제출 화면

## U10. 만료 화면
### 목적
- 유효기간 종료 후 사용자에게 재검증을 유도하는 화면

### 표시 요소
- 상태: `만료됨`
- 만료일
- 재검증 필요 안내

### 버튼
- `재검증 시작`
- `닫기`

### 다음 화면
- U3 제출 화면

## U11. 철회 확인 모달
### 목적
- 공개 중 또는 검증 완료 상태를 사용자가 직접 철회하는 흐름

### 표시 요소
- 경고 문구:
  - `철회하면 프로필에서 즉시 숨김 처리됩니다.`
  - `원하면 나중에 다시 재신청할 수 있습니다.`

### 버튼
- `철회`
- `취소`

### 서버 처리
- 상태 `revoked` 또는 공개만 끄는 정책에 따라 처리
- `is_public = false`
- audit log `badge_revoked`

## 시스템 자동 플로우

### S1. 만료 크론
### 주기
- 하루 1회 이상

### 대상
- `verified` + `expires_at < now()`

### 처리
- badge 상태 `expired`
- 공개 중이면 프로필 숨김
- audit log `badge_expired`

## S2. 원본 파일 삭제 작업
### 트리거
- 승인
- 반려
- 철회

### 처리
- private bucket 파일 삭제
- 삭제 성공 로그 기록
- 실패 시 관리자 알림

## 예외 플로우

### E1. 파일 업로드 실패
- 사용자에게 즉시 에러 표시
- submission 생성 금지

### E2. 중복 제출
- `pending` 또는 `under_review` 가 있으면 새 제출 차단
- 문구: `현재 검수 중인 요청이 있습니다.`

### E3. 승인 후 공개 off
- badge 상태는 `verified` 유지
- `is_public = false`
- 프로필에는 숨김

### E4. 반려 후 재제출
- 새 submission 생성
- badge 상태 `pending`

### E5. 허위 제출 적발
- badge `revoked`
- 운영정책에 따라 계정 제재 연동

## 화면 간 연결 요약
- U1 -> U2 -> U3 -> U4 -> U5
- U5(verified) -> U6 -> U7 -> U8
- U5(rejected) -> U9 -> U3
- U5(expired) -> U10 -> U3
- U5(verified/public) -> U11
- A2 -> A3 -> A4 또는 A5
- A1/A2/A6 -> A7

## 개발 메모
- 사용자 상태 조회는 badge 기준으로 렌더링
- 제출 이력은 submission 기준으로 렌더링
- 프로필 배지는 badge 조건이 맞을 때만 노출
- 검증과 공개 토글은 절대 같은 액션으로 묶지 않음
