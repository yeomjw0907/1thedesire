# 욕망백서 MCP 및 Skill 활용 전략 v0.1

## 1. 목적

- 실제 구현 담당자가 MCP와 Skill을 사용할 수 있을 때 어디에 활용하면 좋은지 정리한다.
- 도구가 있을 때만 쓰고, 없을 때도 흔들리지 않도록 기준을 만든다.

## 2. 기본 원칙

- MCP는 `없어도 구현 가능한 흐름` 위에 얹는 가속 장치로 본다
- Skill은 `반복적으로 같은 맥락을 설명해야 할 때` 만든다
- 문서가 기준이고, MCP/Skill은 실행 보조다

## 3. 현재 판단

현재 이 워크스페이스 문서 기준으로 가장 유용한 MCP 범주는 아래다.

- Supabase MCP
- Filesystem / Repo MCP
- GitHub MCP
- Browser / Playwright MCP
- Notion MCP

단, 어떤 MCP가 실제로 Claude 환경에 연결되어 있는지는 세션마다 다를 수 있다.

## 4. MCP별 활용 포인트

### 4-1. Supabase MCP

가장 유용한 작업:

- 테이블 생성
- 마이그레이션 적용
- RLS 정책 반영
- SQL 실행 결과 확인
- 테스트 데이터 삽입

욕망백서에서 특히 유용한 영역:

- `profiles`, `chat_rooms`, `point_transactions` 테이블 생성
- DM 상태 전이 확인
- 환불 로그 검증
- RLS 디버깅

사용 원칙:

- 문서 기준을 먼저 읽고 SQL을 실행할 것
- 숫자 정책은 문서 기준을 절대 바꾸지 말 것

### 4-2. Filesystem / Repo MCP

가장 유용한 작업:

- 코드베이스 탐색
- 파일 읽기
- 구현 위치 파악
- 반복 파일 수정

욕망백서에서 유용한 영역:

- 프론트 구조 파악
- 서버 액션 위치 찾기
- 상태 관련 유틸 정리

### 4-3. GitHub MCP

가장 유용한 작업:

- PR 작성
- 이슈 정리
- 변경 요약

욕망백서에서 유용한 영역:

- 구현 단위 PR 분리
- 정책 숫자 변경 추적

### 4-4. Browser / Playwright MCP

가장 유용한 작업:

- UI 동작 확인
- 가입 / DM 흐름 테스트
- 모바일 레이아웃 점검

욕망백서에서 유용한 영역:

- 가입 -> 자동 소개글 -> 홈 진입
- DM 요청 -> 거절 / 만료 / 수락 테스트
- 상태 배너와 포인트 반영 검증

### 4-5. Notion MCP

가장 유용한 작업:

- 현재 docs 내용을 외부 운영 문서로 이관
- 팀 공유용 정리

욕망백서에서 유용한 영역:

- 정책 문서 공유
- 운영 매뉴얼 공유
- 출시 체크리스트 공유

## 5. Skill이 특히 유용한 경우

### 5-1. 구현 시작용 Skill

추천 이름 예시:

- `yokmang-implementation`

용도:

- Claude가 욕망백서 구현 시 항상 같은 문서 우선순위와 정책 숫자를 따르게 하기

Skill에 들어가면 좋은 것:

- 핵심 정책 숫자
- 상태값
- 문서 우선순위
- 구현 순서

### 5-2. 운영 검토용 Skill

추천 이름 예시:

- `yokmang-ops-review`

용도:

- 운영 정책, 신고 시나리오, 제재 기준을 빠르게 적용하게 하기

### 5-3. UI 구현용 Skill

추천 이름 예시:

- `yokmang-ui-builder`

용도:

- 디자인 시스템, 카피 톤, 와이어프레임 기준을 반복적으로 지키게 하기

## 6. 지금 가장 현실적인 추천

### 문서만으로도 가능한 것

- Claude 프롬프트에 문서 우선순위 박기
- MCP가 있으면 어떤 도구를 어디에 쓸지 명시하기

### 추가로 만들면 좋은 것

- `yokmang-implementation` Skill 1개

가장 먼저 넣을 내용:

- 새 DM 요청 90P
- 거절 45P 환불
- 24시간 만료 90P 환불
- 차단 환불 없음
- 여성 가입 270P
- PWA 우선
- 홈은 탐색 / 프로필은 판단 / DM은 요청 절차

## 7. Claude 프롬프트에 넣으면 좋은 MCP 지시문

추천 문구:

- 사용 가능한 MCP가 있으면 우선 현재 세션에서 연결된 MCP 목록을 확인하라
- Supabase MCP가 있으면 SQL 수동 작성만 하지 말고 스키마 반영과 검증에 활용하라
- Browser 또는 Playwright MCP가 있으면 가입, DM 요청, 환불 상태 UI를 실제로 검증하라
- GitHub MCP가 있으면 변경사항을 PR 단위로 요약하라
- MCP가 없으면 문서 기준으로 로컬 구현을 진행하라

## 8. 주의할 점

- MCP가 있다고 문서를 건너뛰면 안 된다
- Supabase MCP가 있다고 정책 숫자를 임의 수정하면 안 된다
- Browser MCP 테스트 결과와 문서 기준이 다르면 문서 우선으로 보고 충돌을 명시해야 한다

## 9. 최종 판단

욕망백서에서는 MCP보다 `문서 기준 유지`가 더 중요하다.

하지만 Supabase, Browser, Filesystem 계열 MCP가 있으면 구현 속도와 검증 정확도가 꽤 올라간다.

Skill은 특히 여러 세션에 걸쳐 같은 정책을 반복적으로 지켜야 할 때 가장 가치가 크다.
